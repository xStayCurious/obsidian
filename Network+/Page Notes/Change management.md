-----------------------------------
CHANGE MANAGEMENT
-----------------------------------

Change management is:
- a structured process for managing changes to systems and networks

-----------------------------------
CORE PURPOSE
-----------------------------------

Change management helps:
- reduce outages
- reduce configuration mistakes
- improve stability
- improve accountability
- ensure changes are documented and approved

-----------------------------------
WHY CHANGE MANAGEMENT EXISTS
-----------------------------------

Uncontrolled changes may cause:
- downtime
- security issues
- configuration conflicts
- failed deployments
- service interruptions

-----------------------------------
REQUEST PROCESS TRACKING
-----------------------------------

Request process tracking refers to:
- documenting and monitoring changes throughout their lifecycle

-----------------------------------
CHANGE REQUESTS
-----------------------------------

A change request is:
- a formal proposal to modify systems or infrastructure

-----------------------------------
WHAT CHANGE REQUESTS MAY INCLUDE
-----------------------------------

Examples:
- firewall rule changes
- switch upgrades
- VLAN modifications
- software updates
- server migrations
- AP deployments

-----------------------------------
CHANGE REQUEST PURPOSE
-----------------------------------

Change requests help:
- document planned work
- provide accountability
- allow approval workflows
- support auditing

-----------------------------------
CHANGE MANAGEMENT PROCESS
-----------------------------------

The guide emphasizes:
- structured tracking of requests and approvals

-----------------------------------
COMMON CHANGE MANAGEMENT STEPS
-----------------------------------

Typical process:
1. Submit request
2. Review request
3. Approve/reject request
4. Test change
5. Schedule implementation
6. Implement change
7. Verify results
8. Document completion

-----------------------------------
DOCUMENTATION
-----------------------------------

Proper documentation includes:
- what changed
- who approved it
- when it occurred
- rollback procedures
- implementation results

-----------------------------------
APPROVAL PROCESS
-----------------------------------

Changes commonly require:
- managerial approval
- technical review
- security review

-----------------------------------
WHY APPROVALS MATTER
-----------------------------------

Approvals help prevent:
- unauthorized modifications
- risky deployments
- accidental outages

-----------------------------------
CHANGE WINDOWS
-----------------------------------

A change window is:
- an approved timeframe for implementing changes

-----------------------------------
WHY CHANGE WINDOWS EXIST
-----------------------------------

Change windows help:
- minimize business disruption
- reduce operational impact
- coordinate maintenance

-----------------------------------
COMMON CHANGE WINDOWS
-----------------------------------

Changes are often scheduled:
- overnight
- weekends
- maintenance periods
- low-usage hours

-----------------------------------
EMERGENCY CHANGES
-----------------------------------

Emergency changes are:
- urgent modifications requiring immediate action

-----------------------------------
EMERGENCY CHANGE EXAMPLES
-----------------------------------

Examples:
- critical vulnerability patching
- outage recovery
- security incident mitigation

-----------------------------------
EMERGENCY CHANGE RISKS
-----------------------------------

Emergency changes:
- may bypass some normal procedures
- increase operational risk

-----------------------------------
ROLLBACK PLANS
-----------------------------------

Rollback plans define:
- how to reverse a failed change

-----------------------------------
WHY ROLLBACKS MATTER
-----------------------------------

If a change fails:
- services may need rapid restoration

-----------------------------------
TESTING
-----------------------------------

Changes should commonly be:
- tested before deployment

-----------------------------------
TESTING PURPOSE
-----------------------------------

Testing reduces:
- outages
- incompatibility
- instability
- deployment failures

-----------------------------------
HIGH-YIELD NETWORK+ FACTS
-----------------------------------

Very important exam facts:

- Change management controls modifications
- Change requests document planned changes
- Approvals reduce unauthorized/risky changes
- Change windows minimize disruption
- Rollback plans restore failed changes
- Testing reduces deployment risk
- Audit trails improve accountability
- Emergency changes address urgent issues

-----------------------------------
COMMON EXAM TRAPS
-----------------------------------

Trap:
Thinking change management slows operations unnecessarily.

Wrong.

Its primary purpose is:
- reducing operational risk

-----------------------------------

Trap:
Thinking emergency changes require no documentation.

Wrong.

Emergency changes should still be:
- documented and reviewed

-----------------------------------

Trap:
Thinking rollback plans are optional.

Wrong.

Rollback capability is critical if changes fail.

-----------------------------------

Trap:
Thinking testing is unnecessary for small changes.

Wrong.

Even small changes can create outages.

-----------------------------------

Trap:
Thinking approvals are only technical.

Wrong.

Approvals may include:
- technical
- management
- security
- compliance review
